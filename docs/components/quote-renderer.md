# Рендерер цитат

`src/render/quote.js` преобразует массив сокращённых Telegram Message в изображение. Компонент не обращается к Telegram напрямую: клиент API и функция загрузки файла внедряются при создании.

## Публичный API

```js
const renderer = createQuoteRenderer({ api, downloadTelegramFile });

await renderer.renderQuotePng(messages);     // Buffer PNG
await renderer.renderStickerWebp(messages); // Buffer WebP для Telegram
```

Обе функции строят изображение заново. Входной массив должен быть непустым и содержать сообщения в желаемом порядке.

## Автор сообщения

`quoteUser(message)` учитывает пересылку в следующем порядке:

1. `forward_origin.type === 'user'` — `sender_user`;
2. `hidden_user` — синтетический пользователь с `sender_user_name`;
3. `chat` — ID и title/username исходного чата;
4. иначе — `message.from`.

Имя собирается из имени и фамилии, затем username, затем `Unknown`. Цвет стабильно выбирается как остаток от числового user ID. Для пользователя без ID обычно используется первый цвет.

## Текст и Telegram entities

Текст разбивается `Intl.Segmenter` на графемы. Для каждой графемы проверяются entities Telegram:

- `bold`;
- `italic`;
- `underline`;
- `strikethrough`;
- `code` и `pre`.

Ссылки, mentions, spoilers и custom emoji специального оформления не имеют. Entity лишь влияет на стиль, текст остаётся исходным.

Emoji сначала загружается как Apple PNG с jsDelivr. При неудаче используется Twemoji SVG. Успешные data URI кешируются в process-level `Map`; отрицательный результат не кешируется. Если оба запроса неуспешны, символ рендерится шрифтом.

## Подбор ширины и перенос строк

Рендерер приближённо оценивает пиксельную ширину каждой графемы по классу символа, размеру и bold. Для размеров шрифта из темы перебираются кандидаты ширины. Оценка штрафует переполнение, пустое место справа, неровность строк, короткую последнюю строку и лишнее число строк.

Перенос предпочитает последний пробел в строке; длинный токен без пробелов режется по графемам. Начальные и конечные пробелы строк удаляются. Табуляция отображается четырьмя неразрывными пробелами.

Это эвристика, а не измерение конкретного шрифта, поэтому редкие письменности и сложные glyph shaping могут отличаться от расчёта.

## Поддерживаемые медиа

Приоритет выбора одного media source:

1. самое большое фото из `message.photo`;
2. статический стикер;
3. thumbnail анимированного/video-стикера;
4. thumbnail video note, отображаемый кругом;
5. thumbnail обычного видео.

Фото и видео кодируются JPEG quality 88, стикеры — PNG. Обычные медиа вписываются максимум в 560×560 без увеличения. Video note обрезается до круга 320×320.

Если файл отсутствует, video/video note/sticker получает текстовый fallback. Для других неподдерживаемых типов пустое сообщение может дать практически пустой bubble.

## Аватары и группировка

Последовательные сообщения одного автора образуют группу. Имя показывается только в первом bubble, аватар — только у последнего. От роли bubble (`single`, `top`, `middle`, `bottom`) зависят радиусы и хвост.

Размер аватара зависит от плотности текста, числа сообщений в группе и типа медиа. Если у Telegram-пользователя нет фото или автора нельзя запросить по ID, создаётся цветной круг с первой буквой имени.

Аватар загружается методом `getUserProfilePhotos`, выбирается наиболее крупный размер первого фото, затем Sharp обрезает его до 104×104 PNG. Между рендерами аватары и media не кешируются.

## Pipeline изображения

```text
Messages
  → quoteTree: абсолютная layout-модель
  → Satori: SVG с встроенными fonts/images
  → Sharp: trim прозрачного поля
  ├→ PNG buffer
  └→ resize inside 512×430 → footer 82 px → WebP quality 88
```

SVG использует пять локальных начертаний Noto Sans. Изображения встраиваются как data URI, поэтому итоговая обработка не зависит от внешних ссылок после построения дерева.

Для sticker WebP изображение не увеличивается. После resize добавляется прозрачный footer; итоговая высота равна высоте вписанного изображения плюс 82 px, а не обязательно 512 px.

## Ошибки и ресурсы

Все загрузки и Sharp/Satori-ошибки пробрасываются вызывающему коду. Одно повреждённое медиа прерывает всю цитату. Сообщения рендерятся последовательно, а строки/emoji внутри одной строки загружаются параллельно через `Promise.all`.

При большом количестве уникальных emoji process-level cache растёт без ограничения. Telegram-команда ограничивает цитату десятью сообщениями, но сам публичный renderer этого ограничения не применяет.
