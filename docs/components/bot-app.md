# Приложение Telegram-бота

Файл `src/bot/app.js` связывает Telegram Bot API, Redis, рендерер цитат, загрузчик внешних видео, сервис аналитики и дни рождения.

## Публичный API

```js
createBotApp({ env = process.env, redis, analytics, mediaDownloader, birthdays })
// → { api, chatAllowed, handleUpdate }
```

- `api(method, payload, options)` вызывает метод Telegram Bot API и возвращает `result`; при `ok: false` бросает ошибку.
- `chatAllowed(chatId)` проверяет allowlist. Пустой список разрешает любой chat ID.
- `handleUpdate(update)` — главный обработчик Telegram Update.

`BOT_TOKEN` читается из `env`. Отдельной ранней проверки токена нет: при отсутствии значения приложение создаст URL с `undefined`, а ошибка проявится на первом API-вызове. Числовой префикс токена используется как `botId`, чтобы исключить самого бота из ежедневного выбора и проверить автора стикера для `/qs`.

## Разбор команд

```js
parseCommand(message)
// → null | { name, count, args }
```

Парсер читает `text`, затем `caption`. Допускаются ASCII-команды из букв и `_`, необязательный `@botname` и произвольная строка аргументов. Поле `args` сохраняет строку без крайних пробелов, что позволяет передать дату в `/birthday`.

Числовой `count` ограничивается диапазоном 1…10. Словесные аргументы поддерживают совместимые формы:

- `/pidor list`, `/pidor reset`;
- `/codeword start`, `/codeword hint`, `/codeword stop`.

После нормализации они превращаются в команды с `_`. Для остальных словесных аргументов имя команды сохраняется, а count равен 1.

## Обработка обновления

`handleUpdate` поддерживает только поля `update.message` и `update.edited_message`. Если присутствуют оба, выбирается `message`.

Порядок действий:

1. проверить chat allowlist;
2. сохранить сообщение в Redis;
3. разобрать команду;
4. для обычного сообщения — записать аналитику, проверить кодовое слово и обработать media URL;
5. для `q/qs/qd` — выполнить операцию с цитатой;
6. команды дней рождения передать в Birthday Service;
7. для остальных команд — обратиться к Analytics Service.

Неизвестные команды молча игнорируются. Команды не входят в статистику, кроме `/pidor`: перед выбором пользователя она явно вызывает `analytics.ingestMessage`, чтобы инициатор появился в таблице `users`; её текст всё равно не даёт слов, поскольку токенизатор исключает строки, начинающиеся с `/`.

## Instagram Reels и TikTok

Для обычного сообщения Bot App вызывает `mediaDownloader.urlsFromMessage`. Каждая найденная ссылка обрабатывается последовательно:

1. Telegram получает `sendChatAction` со значением `upload_video`;
2. `mediaDownloader.downloadVideo(url)` возвращает MP4 в Buffer;
3. `sendVideo` отправляет multipart-файл с `supports_streaming=true` как reply на сообщение со ссылкой.

Ошибка одной ссылки сопровождается пользовательским сообщением и не мешает обработать следующие ссылки. Детальная ошибка, код media-сервиса и URL записываются в server log. Ссылки в распознанных командах не обрабатываются, поскольку синтаксис команды не допускает дополнительный URL.

## Redis timeline

Для каждого разрешённого сообщения сохраняется сокращённый payload:

```text
message_id, date, from, forward_origin,
text, caption, entities, caption_entities,
photo, sticker, video_note, video
```

Ключ сообщения: `chat:<chatId>:message:<messageId>`. Он живёт 90 дней.

Timeline: list `chat:<chatId>:timeline`. Новый ID добавляется слева, list обрезается до 10 000 элементов и также живёт 90 дней. Edited message может добавить повтор ID; при чтении повторы удаляются через `Set`.

`collectMessages(chatId, startId, count, beforeId)` читает до 10 000 ID, выбирает диапазон `startId <= id < beforeId`, сортирует по возрастанию, по одному загружает сообщения и исключает распознанные команды. Возвращается не более `count` элементов.

## Создание цитаты: `/q [N]`

Команда должна отвечать на первое цитируемое сообщение. Reply принудительно записывается в Redis, что полезно, если его не было в timeline ранее. Затем сообщения собираются до ID команды и передаются `renderStickerWebp`.

Готовый buffer отправляется через `sendSticker` как multipart поле `sticker`, имя файла `quote.webp`, с ответом на сообщение-команду.

Без Redis `collectMessages` всегда пуст, поэтому даже `/q` для одного reply завершится подсказкой о кеше.

## Стикерпак: `/qs` и `/qd`

`/qs` принимает только reply на статический, не video и не animated стикер, отправленный текущим ботом. Файл скачивается из Telegram.

Сначала вызывается `addStickerToSet`. При любой ошибке создаётся новый набор через `createNewStickerSet`; затем пользователю отправляется Markdown-ссылка на набор. Для обоих методов владельцем указывается пользователь, вызвавший команду.

`/qd` требует reply на любой стикер и вызывает `deleteStickerFromSet` с его `file_id`. Приложение не проверяет имя набора или автора; Telegram отклонит запрещённую операцию.

## Telegram API и файлы

JSON-вызовы получают `content-type: application/json`. Файловые вызовы используют `FormData` и не задают заголовок вручную, чтобы runtime добавил boundary.

`downloadTelegramFile(fileId)` выполняет два запроса: `getFile`, затем GET к `https://api.telegram.org/file/bot<TOKEN>/<file_path>`. Результат полностью загружается в `Buffer`.

## Ошибки и производительность

Внутри Bot App ошибки не перехватываются, кроме fallback создания sticker set. Они доходят до HTTP-входной точки. Отправка серии сообщений и загрузка сообщений из Redis выполняются последовательно. Это сохраняет порядок ответов, но увеличивает время обработки `/pidor` и `/q N`.
