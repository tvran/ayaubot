# Архитектура

## Общая схема

```text
Telegram
   │ HTTPS Update
   ▼
Vercel handler ─┐
                ├─► Bot App ─► Telegram Bot API
Node HTTP server┘       │
                       ├─► Redis: кеш сообщений
                       ├─► Analytics Service ─► PostgreSQL
                       ├─► Media Service ─► yt-dlp + ffmpeg
                       └─► Quote Renderer
                              ├─► Telegram File API
                              ├─► CDN с emoji
                              └─► Satori + Sharp
```

Обе входные точки один раз при загрузке модуля создают PostgreSQL-подключение, сервис аналитики, Media Service и приложение бота. Обработчик конкретного HTTP-запроса передаёт Telegram Update в `bot.handleUpdate()`.

## Компоненты и ответственность

| Компонент          | Файлы                                    | Ответственность                                                 |
| ------------------ | ---------------------------------------- | --------------------------------------------------------------- |
| Входные точки      | `api/telegram.js`, `src/server/index.js` | HTTP-маршрутизация, проверка секрета, сборка зависимостей       |
| Bot App            | `src/bot/app.js`                         | Допуск чатов, команды, Telegram API, Redis timeline, стикерпаки |
| Analytics Service  | `src/analytics/service.js`               | Пользовательские сценарии статистики и игр                      |

| PostgreSQL adapter | `src/db/postgres.js`                     | Инициализация схемы и SQL-операции                              |

## Поток обработки Update

1. Входная точка принимает JSON и при заданном `WEBHOOK_SECRET` проверяет заголовок `x-telegram-bot-api-secret-token`.
2. `handleUpdate` выбирает `message` или `edited_message`. Остальные типы обновлений игнорируются.
3. Сообщения из чатов вне allowlist игнорируются до кеширования и аналитики.
4. Допущенное сообщение сохраняется в Redis timeline, если Redis настроен.
5. Если текст не распознан как команда, сервис обновляет статистику, проверяет угадывание кодового слова и обрабатывает Reels/TikTok URL.
6. `/q`, `/qs` и `/qd` направляются в подсистему цитат; остальные известные команды — в аналитику.
7. Неизвестная команда не получает ответа и не попадает в статистику слов.

Входные точки намеренно отвечают HTTP 200 даже при внутренней ошибке обработки. Это предотвращает бесконечные повторы Telegram, но означает, что потерянное обновление можно обнаружить только по логам.

## Поток создания цитаты

Команда `/q N` должна быть ответом на первое сообщение. Ответное сообщение принудительно кешируется, затем из Redis выбираются сообщения с `message_id` от него включительно до сообщения-команды исключительно. Команды исключаются, ID сортируются по возрастанию, результат ограничивается `N` сообщениями.

Рендерер определяет фактического автора с учётом `forward_origin`, загружает аватар и медиа через Telegram API, строит дерево Satori, получает SVG и передаёт его Sharp. Итоговый WebP вписывается в область 512×430 без увеличения и получает прозрачный нижний отступ 82 px.

## Поток аналитики

Обычное сообщение сначала обновляет профиль пользователя в таблице `users`. Затем текст или caption очищается от URL, токенизируется и агрегируется в `word_counts` по чату, пользователю, слову и календарному дню.

Игры используют накопленные данные:

- кодовое слово случайно выбирается из топ-15 слов за 14 дней и действует три дня;
- подсказка использует лидера по выбранному слову за 14 дней;
- ежедневный выбор берёт случайного известного участника и сохраняет результат на текущую дату.

## Поток загрузки внешнего видео

Media Service извлекает из text/caption до трёх уникальных поддерживаемых URL. Для каждой ссылки Bot App показывает действие `upload_video`, а сервис запускает отдельный `yt-dlp` без shell. Видео скачивается во временную директорию, при необходимости объединяется или перекодируется ffmpeg в MP4, читается в память и отправляется методом Telegram `sendVideo` как reply. Временная директория удаляется в `finally` и при успехе, и при ошибке.

## Границы состояния

- Процесс Node.js хранит только кеш загруженных emoji (`Map`) и созданные объекты сервисов.
- Redis хранит сокращённые Telegram-сообщения 90 дней и timeline максимум из 10 000 записей.
- PostgreSQL хранит профили, агрегаты слов, игры и ежедневные выборы без автоматического срока удаления.
- Telegram хранит исходные файлы и стикерпаки.
- Временная директория ОС кратковременно хранит загружаемые видео; после каждого вызова она рекурсивно очищается.

Отсутствие Redis не мешает аналитике, но `/q` не сможет получить даже ответное сообщение: `collectMessages` возвращает пустой список. Отсутствие PostgreSQL отключает хранение данных; сервис возвращает пользователю сообщение о ненастроенной аналитике.

## Внешние зависимости

- Telegram Bot API и Telegram File API;
- Upstash Redis через REST;
- PostgreSQL через `pg`;
- jsDelivr для Apple emoji и резервного Twemoji;
- Satori для SVG-разметки;
- Sharp/libvips для обработки изображений.
- системные `yt-dlp` и `ffmpeg` для внешних видео.

## Существенные ограничения текущей реализации

- Автоматические тесты покрывают URL-фильтр и базовый lifecycle Media Service; остальные компоненты тестами не покрыты. `npm run lint` выполняет только синтаксическую проверку файлов.
- У API Telegram, загрузки файлов и CDN emoji нет таймаутов или повторов.
- Одновременные вызовы ежедневного выбора могут вернуть разным запросам разных кандидатов: вставка защищена `ON CONFLICT`, но проигравший запрос не перечитывает сохранённую строку.
- Одновременные угадывания кодового слова могут сформировать несколько победных ответов, поскольку результат условного `UPDATE` не проверяется.
- Любая ошибка `addStickerToSet` трактуется как отсутствие набора и запускает `createNewStickerSet`, что может скрыть исходную причину.
- `/qd` передаёт Telegram любой `file_id` стикера без локальной проверки владельца набора; фактический контроль выполняет Telegram.
- Загрузка шрифтов зависит от `process.cwd()`, который должен указывать на корень проекта.
- Загрузка внешнего видео держит готовый файл в `Buffer` до окончания multipart-запроса, поэтому лимит файла должен учитывать доступную память процесса.
